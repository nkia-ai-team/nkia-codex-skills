# Linear 작업 템플릿별 상세 가이드

모든 이슈는 가이드라인의 6섹션 번호 구조를 공통으로 사용합니다.
공통 템플릿 구조는 [guideline-ref.md "이슈 템플릿"](../../_shared/guideline-ref.md) 참조.

작업 유형별로 각 섹션에 어떤 내용을 채워야 하는지 아래에서 안내합니다.

---

## 작업 유형별 섹션 내용 가이드

### 1. 빌드/배포 (Task 타입)

**자동 라벨:** "build"

| 섹션 | 작성 가이드 |
|------|-----------|
| 1. 문제/배경 | 왜 이 빌드/배포가 필요한지 |
| 2. 목표/기대 결과 | 무엇을 빌드/배포하는지, 대상 환경 |
| 3. 완료 조건 | 배포 체크리스트 (Dockerfile, Jenkinsfile, 빌드 성공, 이미지 배포 등) |
| 4. 범위 | 대상 서비스, 환경 (dev/staging/prod) |
| 5. 검증 방법 | Healthcheck, 버전 확인, API 호출 테스트 |
| 6. 참고 자료 | 이미지 태그, 환경 설정, 관련 이슈 |

**수집 정보:**
- 배경 (왜 필요한지)
- 작업 설명 (무엇을 빌드/배포하는지)
- 배포 체크리스트 항목 (선택)
- 참고사항 (선택 - 이미지 태그, 환경 설정, 관련 이슈)

## AC 생성 패턴 (Template-Specific)

> 원칙: AC 3~5개 이내로 유지. 가장 중요한 검증 포인트만 포함.

**⚠️ CRITICAL: AC 결과물 원칙**

AC 결과물은 **"코드를 변경했다"가 아니라 "변경한 코드가 의도대로 동작한다"를 증명**해야 합니다.

**금지하는 증빙 유형:**
- `코드 변경`, `git diff`, `diff --stat` — 코드를 바꿨다는 것은 PR/MR에서 이미 확인됨
- `코드 리뷰 통과` — 공통 AC에서 처리됨
- `구현 완료`, `수정 완료` — 동작 증명이 아님

**허용하는 증빙 유형 (실제 실행 결과만):**

| 증빙 유형 | 설명 | 예시 |
|-----------|------|------|
| 테스트 결과 | 단위/통합/E2E 테스트 실행 출력 | `pytest -v` 전체 출력 |
| 실행 로그 | 애플리케이션/서버/Docker 실행 로그 | `docker logs` grep 결과 |
| 스크린샷 | UI 동작 화면 캡처 | Playwright FHD 스크린샷 |
| API 응답 | curl/HTTP 클라이언트 실행 결과 | `curl -s` 응답 본문 |
| DB 쿼리 결과 | 실제 DB 조회 출력 | `mongosh`/`psql` 쿼리 결과 |
| 메트릭 결과 | 평가 스크립트 실행 출력 | 정확도/성능 수치 |
| CI/CD 로그 | 빌드/배포 파이프라인 결과 | `gh run view` 로그 |
| 네트워크 캡처 | SSE/WebSocket/HTTP 이벤트 로그 | 이벤트 스트림 캡처 |
| 데이터 경로 | 파일 존재 + 건수 + 샘플 | `ls` + `wc -l` + `head` |

**빌드/배포:**
- [ ] CI 빌드 성공 및 이미지 배포 완료 → 결과물: 빌드 로그 {{ci_log_link}}

**데이터 작업:**

## 제목 개선 가이드라인

무엇을 왜 하는지 한 줄로 파악 가능하게 작성

### 포괄적 제목 작성 원칙

**⚠️ 제목이 특정 모듈/스킬/파일에 한정되지 않도록 주의할 것!**

이슈의 작업 범위가 여러 모듈에 걸칠 수 있으므로, 제목은 작업의 **목적이나 카테고리**를 포괄적으로 표현해야 합니다. 특정 대상에 한정된 제목은 같은 성격의 수정을 다른 대상에도 적용할 때 별도 이슈를 만들어야 하는 불편함을 초래합니다.

| 상황 | Bad (한정적) | Good (포괄적) |
|------|-------------|--------------|
| 여러 스킬 수정 | `linear-issue-validator 이미지 검증 개선` | `검증·생성 스킬 동작 개선` |
| 여러 API 수정 | `UserController 인증 오류 수정` | `인증 API 토큰 검증 로직 수정` |
| 여러 컴포넌트 수정 | `Sidebar 다크모드 스타일 적용` | `다크모드 스타일 일괄 적용` |

**적용 기준:**
- 범위(Scope)에 2개 이상 대상이 명시되면 → 포괄적 제목 사용
- 범위가 단일 대상이면 → 구체적 제목 유지

**빌드/배포:**

## 작업 템플릿 → 이슈 타입/라벨 자동 매핑

| 작업 템플릿 | Linear 이슈 타입 | 자동 라벨 (Work type) |
|------------|----------------|---------------------|
| 빌드/배포 | Task | "build" |
| 데이터 작업 | Task | "data" |
| 평가 | Task | "research" |
| 새로운 기능 개발 | Feature | "feature" |
| 기능 개선 | Feature | "improve" |
| 리팩토링 | Feature | "refactor" |
| 리서치 | Research | "research" |
| 버그 수정 | Bug | "bug" |
| 문서 작업 | Task | "document" |

### 라벨 체계

라벨은 **작업 성격(Work type)** 과 **도메인(Domain)** 으로 나뉘며, 복수 라벨 조합이 가능합니다.

| Category | Labels | 설명 |
|----------|--------|------|
| Work type | bug, feature, improve, refactor, research, document, task | 작업의 성격 (what) — 템플릿 선택 시 자동 부여 |
| Domain | build, infra, data | 작업의 대상/영역 (where) — 내용 분석을 통해 추가 부여 |

### 내용 기반 도메인 라벨 추가:
이슈 제목/설명을 분석하여 해당하는 도메인 라벨을 추가 부여합니다:
- "build" — CI/CD, 빌드 파이프라인, 배포 관련
- "infra" — 서버 설정, 환경변수, compose/env 관리, 인프라 아키텍처 관련
- "data" — 데이터 수집/가공/마이그레이션 관련

**복수 라벨 조합 예시:**
- 데이터 파이프라인 리팩토링 → "refactor" + "data"
- 배포 문서 작성 → "document" + "build"
- 온보딩 매뉴얼 리뉴얼 → "document" + "improve"
- Redis config 환경변수화 → "improve" + "infra"
