# Report Rendering

수집된 데이터를 시트 컬럼별 형식에 맞게 렌더링하는 규칙을 정의합니다.

---

## 1. 입력 데이터 정리

렌더링 전에 수집된 데이터를 **프로젝트 단위로 그룹핑**합니다.

    grouped = {
      "RCA Portal 연동": {
        done: [NKIAAI-414, NKIAAI-413, NKIAAI-380, ...],
        in_review: [NKIAAI-433, ...],
        in_progress: [NKIAAI-433],
        commits: { "lucida-chat-ai": [...], "lucida-ui": [...] }
      },
      "RCA Agent": {
        done: [NKIAAI-311, NKIAAI-420],
        in_progress: [],
        commits: { "rca-agent": [...] }
      }
    }

프로젝트가 없는 이슈는 "기타" 그룹으로 분류합니다.

---

## 2. B열 — 업무구분 (금주)

**기본값: `백로그`**

드롭다운 값이므로 시트에 존재하는 값과 정확히 일치해야 합니다.

---

## 3. C열 — 업무 (목표일, 진행율)

이슈를 개별로 나열하지 않고, **관련 이슈를 큰 작업 주제로 묶어서** 번호 리스트로 작성합니다.

### 형식

    1. {큰 작업 주제}
    2. {큰 작업 주제}
    3. 기타

### 그룹핑 규칙

- Done/In Review 이슈를 금주 실적의 주 소스로 사용하고, 같은 기능/목적에 해당하는 이슈들을 하나의 주제로 묶기
  - 예: "멀티턴 후속 질문", "멀티턴 컨텍스트 보강", "멀티턴 팝업 렌더링" → **"RCA 멀티턴 대화 기능 구현"**
  - 예: "Workspace TTL 정리", "Celery 로그 수정", "통합 테스트 구축" → **"RCA Agent 인프라 개선"**
- 묶을 수 없는 단독 작업은 그대로 하나의 항목으로
- 소규모 작업(버그 수정, 스타일 수정 등)은 "기타"로 통합
- 휴가/반차가 있으면 "기타"에 포함
- **1~5개 항목**이 적정. 실제 작업이 하나면 1개만 작성합니다.
- Feature/프로젝트 이름 수준으로 쓰고, 구현 세부사항은 D열에 둡니다.
- 이슈 번호, PR 번호, Linear 상태명은 C열에 쓰지 않습니다.

### 예시

    1. RCA 테스트베드 자동화 스킬 dogfooding 9 라운드 진행
    2. testbed-build 오케스트레이터 재설계
    3. testbed-services 도메인 추가 (food-delivery, social-feed)

---

## 4. D열 — 업무 내용 (상세)

C열의 각 번호 항목에 대한 **상세 내용**을 기술합니다. 이 컬럼이 보고서의 핵심입니다.

### 형식

    1. {C열의 같은 번호 제목}
     - {상세 작업 내용 1}
     - {상세 작업 내용 2}
     - {상세 작업 내용 3}

    2. {C열의 같은 번호 제목}
     - {상세 작업 내용 1}
    ...

번호 제목은 C열과 최대한 동일하게 유지합니다. 서브불릿은 하이픈 앞에 공백 하나를 둔 ` - ` 형식을 사용합니다.

### 상세 내용 소스 우선순위

1. **커밋 메시지** — 해당 이슈에 매핑된 커밋의 설명에서 추출
2. **이슈 description의 체크된 AC** — `[X]` 항목에서 달성 내용 추출
3. **이슈 제목** — 위 소스가 없으면 제목만 기술

### 커밋 기반 상세 내용 추출

커밋 메시지에서 의미 있는 변경 사항을 추출합니다:

    # 커밋 메시지 예시
    nkiaai-414 Improve : RCA 멀티턴 후속 질문 시 alarm_select_popup 대신 LLM 기반 답변 생성

    → 추출: "RCA 멀티턴 후속 질문 시 alarm_select_popup 대신 LLM 기반 답변 생성"

동일 이슈의 커밋이 여러 개이면 각각을 서브불릿으로 나열하되, 중복되는 내용은 병합합니다.

### AC 기반 상세 내용 추출

이슈 description에서 체크된 AC 항목을 추출합니다:

    - [X] RCA 분석 완료 후 후속 질문 시 alarm_select_popup 대신 텍스트 답변이 스트리밍되는 것 확인
    → 추출: "RCA 후속 질문 시 텍스트 스트리밍 확인"

AC의 `→ 결과물:` 이후 부분과 코드 블록(```)은 제거하고 핵심 내용만 추출합니다.

### 휴가/반차 반영

캘린더에서 수집된 휴가/반차 이벤트는 "기타" 항목에 포함합니다:

    5. 기타
     - 4/7(월) 오전반차
     - 4/8(화) 연차

### 예시 (전체)

    1. RCA 테스트베드 자동화 스킬 dogfooding 9 라운드 진행
     - test 누적 피드백 반영해 1.9.23 → 1.9.39 까지 9회 패치 릴리즈
     - 자격증명 인터뷰 흐름, Polestar10 등록 안정화, 알람 정책 일관화 등 사용자 흐름 위주 개선
     - testbed 별 K8s cluster 격리(k3d) 도입으로 동시 다발 testbed 운영 가능

    2. testbed-build 오케스트레이터 재설계
     - 9 라운드 dogfooding 회고 결과 monolithic 구조 한계 확인 → sub-agent dispatch 구조로 재설계
     - SKILL.md 슬림화로 context 사용량/실행 시간 감소

    3. testbed-services 도메인 추가
     - food-delivery, social-feed 두 도메인 scaffold 추가 — testbed 시연/검증용 다양화

## 5. F열 — 업무 구분 (차주)

**기본값: `백로그`**

B열과 동일한 규칙입니다.

---

## 6. G열 — 차주 업무

다음 사이클 또는 Todo 상태의 이슈를 기반으로 차주 계획을 작성합니다.

### 형식

    1. {차주 큰 작업 주제}
     - {구체적으로 할 작업}
     - {구체적으로 할 작업}

    2. {차주 큰 작업 주제}
    ...

단순 계획은 서브불릿 없이 번호 리스트만 작성해도 됩니다.

### 소스

1. **In Progress 이슈** — 이번 주에 완료하지 못한 진행 중 이슈 (최우선)
2. **Todo 이슈** — 다음 사이클/Todo 상태 이슈
3. **이슈 description** — "목표/기대 결과" 또는 미체크 AC에서 할 작업 추출

### 구체적 작업 내용 추출

- In Progress 이슈: **미체크 AC(`- [ ]`)** 항목에서 남은 작업 추출
- Todo 이슈: **"목표/기대 결과(What)"** 섹션에서 핵심 작업 추출
- 둘 다 없으면 이슈 제목만 기술

### 예시

    1. testbed-build 오케스트레이터 스킬 1차 완성
    2. 장애 시나리오 50개 작성
    3. AI portal RCA UI 연동

또는 상세 계획이 필요하면:

    1. query_itg_mongo 자연어 시간 범위 필터 구현
     - "최근 일주일/한 달" 등 자연어 시간 표현 → date 필터 변환 로직 구현
     - LLM 잘못된 단정 응답 회귀 테스트

## 7. 렌더링 품질 규칙

### 글쓰기 원칙

- **한글로 작성** — 기술 용어(API, MR, JWT 등)는 영문 그대로 사용
- **간결하게** — 서브불릿은 1줄, 불필요한 설명 생략
- **구체적으로** — "기능 개선" 대신 "alarm_select_popup 대신 LLM 기반 답변 생성"
- **동사로 시작** — "구현", "추가", "수정", "설계", "확인" 등

### 분량 가이드

| 컬럼 | 권장 분량 |
|------|----------|
| C열 | 1~5개 항목 |
| D열 | 항목당 1~4개 서브불릿 |
| G열 | 1~5개 항목, 필요 시 항목당 1~3개 서브불릿 |

### 생략 규칙

- 1 Point 이하의 사소한 수정(typo, 스타일 변경)은 "기타"로 묶기
- 동일 프로젝트의 연관 이슈(예: A 이슈 완료 후 B 이슈에서 후속 작업)는 맥락을 연결하여 기술
- 커밋 메시지가 "WIP", "fix typo" 등 의미 없는 경우 건너뛰기
- C/G에는 과도한 하위 bullet을 넣지 말고 D/G 상세가 필요한 경우에만 사용
- "완료", "진행중", "착수" 같은 상태 단어는 필요할 때만 자연스럽게 사용하고 Linear 상태명을 그대로 나열하지 않기
